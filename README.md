# Сейсмостанция «21M» v2.12.0

Консервативный режимный дэшборд Bitcoin для GitHub Pages. Он не предназначен для интрадей-трейдинга
и не пытается предсказать цену. Панель отвечает на два практических вопроса:

1. **Краткосрочно, дни–недели:** устойчив ли путь цены или он хрупок из-за плеча, волатильности и
   слабого спот-подтверждения?
2. **Среднесрочно, недели–месяцы:** согласованы ли глобальный режим, маржинальный спрос и цикл
   Bitcoin настолько, чтобы сохранять или увеличивать экспозицию?

40 карточек — 32 в решающем контуре (20 голосующих семей, 5 блоков) и 8 теневых без голоса.
Всё считается из бесплатных источников, без ключей (кроме опционального FRED-ключа) и без
npm-зависимостей.

## Развёртывание

Подробная пошаговая инструкция — в `РАЗВЁРТЫВАНИЕ.md`. Кратко:

1. Публичный репозиторий → **Settings → Actions → General → Workflow permissions → Read and write**.
2. **Settings → Pages → Source → GitHub Actions** (НЕ «Deploy from a branch»).
3. Загрузить файлы.
4. **Actions → Обновление Bitcoin-снимка → Run workflow.**
5. Сайт: `https://<логин>.github.io/<репозиторий>/`

Ключи необязательны: `FRED_KEY` ускоряет макро-блок, `CM_API_KEY` повышает шанс получить биржевые
потоки. Без них панель работает — FRED читается через keyless CSV-эндпоинт.

### Почему источник Pages должен быть «GitHub Actions»

Документация GitHub: *«Commits pushed by a GitHub Actions workflow that uses the GITHUB_TOKEN do not
trigger a GitHub Pages build.»* При источнике «Deploy from a branch» коммит снимка **не пересобрал бы
сайт**: Action был бы зелёным, данные легли бы в репозиторий, а страница вечно показывала бы старый
снимок. Поэтому workflow публикует сайт сам (`upload-pages-artifact` → `deploy-pages`), а коммит
остаётся только ради истории. `workflow-test` следит, чтобы эти шаги не пропали.

Снимок обновляется раз в час. Живой кандидат собирается во временные файлы и попадает в `docs/`
только после строгой проверки — на сайте всегда остаётся последний подтверждённый снимок.

### Внешний планировщик и сторож (критически важно для непрерывности)

Штатный `schedule:` GitHub Actions — best-effort и фактически отдаёт **2–6 тиков из 24 в сутки**
(наблюдалось окно 6.5 ч без единого запуска). Поэтому публикацию ежечасно (в **:50**) дублирует
внешний джоб на **cron-job.org**: POST на
`/repos/<owner>/<repo>/actions/workflows/snapshot.yml/dispatches` с телом `{"ref":"main"}` и
fine-grained PAT (**только Actions: write, только этот репозиторий**). **PAT истекает ~в середине
октября 2026**: симптом протухания — `401` в cron-job.org и замёрзшая страница; лечение — выпустить
новый токен и вписать его в тот же джоб. Родной cron остаётся бесплатным резервом.

Сторож (`monitor.yml`, каждые 2 часа) страдает от того же планировщика, поэтому: (1) проверка
живого контура встроена **в каждый ежечасный прогон** (не блокирует публикацию, синхронизирует
инцидент-issue), (2) `monitor.yml` принимает `workflow_dispatch` — рекомендуется второй джоб на
cron-job.org (например, в :20 каждые 2 часа) на `monitor.yml/dispatches` тем же PAT.

### Лестница источников ETF

Потоки spot-ETF — главная нога блока спроса, поэтому у неё самый сложный тракт. Раньше это
описание висело в карточке на панели и занимало пол-экрана; логика живёт здесь и в комментариях
`fetchEtfFlows` / `spliceFreshEtfDays`.

**Канон — The Block.** У него два зеркала с побайтово идентичными рядами (628 общих дней), но
зеркало `data.tbstat.com` обновляется раньше. Берётся не «первый успешный», а **самый свежий
валидный завершённый** источник; если зеркала разошлись на общих днях, побеждает канонический
chart-API, а расхождение уходит в диагностику.

**SosoValue дополняет, а не заменяет.** Он добавляет только те торговые дни, которых у канона ещё
нет, и вытесняется, как только канон их догоняет. Ограничения дополнения:

- **Подтверждение вторым наблюдением.** Свежий день берётся, только если прошлый часовой прогон
  видел у него ровно то же значение. Причина: полноту дня у провайдера спросить нельзя — все фонды
  всегда несут одну дату публикации, а неотчитавшийся фонд выглядит ровным нулём, а не отставшей
  датой. Зато неполный день **дорастает** по мере отчёта фондов, то есть меняется между
  наблюдениями. Цена — час задержки на самом свежем дне.
- **Сверка на пересечении.** Если провайдеры расходятся на общих днях больше чем на 3%, сшивка
  отменяется целиком: это признак смены методологии или единицы измерения.
- **Предел опережения — 3 дня**, и он считается по сырому опережению, до любого отсева. Больше —
  это уже подмена источника, а не дополнение.
- **Кэш подтверждённого дня.** Разовый отказ провайдера не выбрасывает из ряда уже подтверждённый день: он переживает сбой сутки от момента подтверждения. Иначе получался бы одиночный встречный снимок, а тот обнуляет 48-часовой отсчёт гистерезиса — на исторической реконструкции пропажа свежего дня переворачивает кандидат среднесрочного режима на 4.2% дней. Кэш проходит те же проверки, что и живые данные, и вытесняется догнавшим каноном.
- **Выравнивание меток времени.** Канон ставит полдень UTC, SosoValue — полночь. Без приведения к
  соглашению канона шаг между последними точками выходит 12 часов вместо суток, а возраст
  наблюдения прыгает на полсуток без единого нового факта. День, чей канонический момент ещё не
  наступил, **отбрасывается**, а не подтягивается к «сейчас»: подтягивание впустило бы незакрытый
  торговый день в окна потоков.

**Три ступени при отказе.** Порядок важен, потому что перцентиль потоков считается по **всей**
глубине ряда, и подмена 630 дней канона на 300 дней резерва сдвигает ранги (на живой точке
p20 15.2 против 10.1 — картина заметно менее медвежья):

1. живые зеркала The Block + дополнение SosoValue;
2. зеркала недоступны → каноном становится **кэш прошлого прогона** (история The Block живёт в
   состоянии неделю), SosoValue дополняет его как обычно. Глубина сохраняется, пакет помечается
   неполным. Каноном из кэша имеет право стать только сам канон: пусти туда прошлый резерв — и
   однажды случившаяся подмена закрепится навсегда, копируясь из прогона в прогон;
3. кэша нет вовсе (первый запуск, потеря кэша, кэш старше недели) → ряд целиком берётся у
   SosoValue, пакет помечается неполным, смещение перцентилей называется в диагностике.

**Farside убран из тракта.** Он отдавал 403 на каждом прогоне (Cloudflare режет раннеры GitHub),
то есть резерва не существовало: при отказе канона панель осталась бы вовсе без потоков ETF. Он
же давал вечный «Critical failures: 1» в диагностике, а постоянный красный приучает её не читать.

### Ловушка, из-за которой контракт существует

Сверено по реально скачанным данным:

```
хешрейт:  Coin Metrics → 1.114e9  (TH/s)     mempool.space → ~7e20  (H/s)
                                              разница в 1e12 раз
```

Тихое сшивание этих рядов дало бы «обвал» дохода майнеров на 1e14 % и запустило бы ветку
капитуляции. Контракт определяет масштаб по последней точке (самой ограниченной), применяет его ко
всему ряду и **отвергает источник при неоднозначности**.

### Два жёстких правила

1. **Никогда не сшивать двух провайдеров в один ряд.** Ряд несёт имя своего провайдера.
   Относительная метрика (изменение за 90 дней), посчитанная через точку смены провайдера,
   бессмысленна, даже если оба провайдера по отдельности правы. Пример: Coin Metrics отдаёт объём по
   **всем** площадкам (7.5e9 USD/день), Coinbase — по своей книге (~1e9 USD/день). Единица одна,
   охват разный, сдвиг уровня в 7 раз.
2. **Источник вне физической полосы отвергается, а не используется.** Неверное число хуже
   отсутствующего.

### Что честно не имеет бесплатной замены

Притворяться, что резерв есть, хуже, чем сказать прямо:

- **Биржевые балансы и потоки** (Coin Metrics `SplyExNtv`, `FlowIn/OutExNtv`) — размеченные биржевые
  адреса бесплатно не публикует никто. При падении Coin Metrics эти карточки гаснут, но требуемую
  группу закрывает `stablecoin_regime`.
- **История предложения стейблкоинов** (DefiLlama) — закрыта той же группой.
- **DVOL и опционный skew** (Deribit) — эквивалента нет; слой необязателен.

Единственная незаменимая точка модели — **дневной ряд цены**, и у неё три независимых источника
(Coinbase, Bitstamp, blockchain.info).

## Теневой слой (vote:false)

Восемь карточек-кандидатов из аудита 2026-07-21 копят **форвардные** доказательства для возможной
policy v2: дневная TGA, G3-ликвидность (ФРС+ЕЦБ+BoJ), корреляция BTC↔золото (PAXG), себестоимость
STH, SOPR/LTH-SOPR, терм-структура IV с put/call OI, суточный сброс OI, hash-ribbons. Правила слоя:

- **строго без голоса** (`vote:false`, `score:null`): ни в баллы, ни в детекторы, ни в аллокацию;
- **отказ источника не деградирует quality решения** (`decision_relevant:false` — решающее правило
  этих данных не читает, поэтому их сбой — не его проблема);
- **право голоса возможно только через policy v2**: предзаявленные критерии, форвардное окно,
  явное решение владельца — никакой калибровки по истории (POLICY.md);
- STH/SOPR перенесены сюда из exclusions решением владельца 2026-07-21; NUPL по-прежнему исключён
  (почти монотонная функция MVRV — valuation считалась бы трижды).

## Гейты вердикта

- **Негативная лестница заякорена на спросе.** Одинокий слабый macro- или cycle-блок даёт лишь
  «переходный» режим; «ухудшается» требует слабого спроса; «защитный» — двух слабых блоков.
  Реконструкция 2019–2026 показала: старое правило «любой один слабый блок → ухудшается» держало
  панель в этом состоянии 36% всех дней с фактической средней форвардной доходностью +20% за 180
  дней — предупреждение без информации, съедавшее весь апсайд 2023–2024.
- **Три детектора имеют право голоса.** «Макрошок ликвидности» (все три макро-семьи в стрессе;
  такие дни в среднем −16% за следующие 90д) ограничивает вердикт сверху; «Капитуляция →
  восстановление» (дешёвый MVRV + ≥2 подтверждения; +10% за 30д, hit 0.82) поднимает
  защитный/ухудшающийся режим до переходного; «Дистрибуция» (дорогой MVRV + рост биржевого
  предложения И слом тренда — требуются оба подтверждения) ограничивает вердикт сверху.
  (Комментарий «Two detectors…» внутри замороженной секции кода устарел — прав голоса три; текст
  запинен вместе с логикой и будет поправлен только в policy v2.)
- **Асимметричный гистерезис.** Переход к защите подтверждается двумя снимками (~2 часа); переход к
  риску дополнительно должен продержаться 48 часов и ≥12 наблюдённых снимков. Плечо и волатильность голосуют только штрафом:
  «нормальные» funding/basis/RV/DVOL — это отсутствие угрозы (0), а не аргумент за покупку.
- **Календарь публикации задаёт порог свежести.** Каждый FRED-ряд объявляет `release`: `daily`
  (новая точка каждый рабочий день → 7 дней), `weekly-batch` (дневные значения приходят одним
  пакетом раз в неделю: H.4.1 по четвергам, H.10 по понедельникам → 14 дней, при праздничном
  переносе последняя точка штатно доживает до 11 дней 20 часов) или `monthly-batch` (JPNASSETS:
  точка датируется первым числом месяца, штатная старость до ~62 дней → 70 дней). Порог обязан совпадать с
  `sourceMaxAgeH` в self-test — unit-test проверяет и связь `release`↔порог, и совпадение таблиц:
  рассинхрон молча отвергал бы валидного кандидата и морозил сайт.
- **Неполная семья предупреждает, но не поддерживает.** Потеряв половину и более своих входов,
  семья не может отдать положительный голос (балл ограничивается нулём), но её предупреждение
  проходит полностью. Семья при этом не обнуляется: `null` отдал бы одному ряду право погасить
  обязательную семью и весь среднесрочный вердикт.
- **Гейт оценки (асимметричный).** MVRV не является жёстким требованием: один вендор не должен уметь
  выключить панель. Но его отсутствие **никогда не создаёт оптимизм** — без данных оценки
  среднесрочный вердикт не поднимается выше «переходного». Предупреждения при этом не подавляются.
  Картина «цена растёт, потоки идут» без оценки цикла — ровно та рефлексивная ловушка, ради которой
  модель и строилась.
- **Тактика не зависит от бирж.** Offshore-площадки (OKX) часто недоступны с американских IP
  раннеров GitHub. Опорой краткосрочного вердикта служит реализованная волатильность, которая
  считается из собственного ценового ряда и не может быть заблокирована геофильтром.
- **Обязательные семьи** для среднесрочного вердикта: ликвидность, финансовые условия, ETF,
  (стейблкоины | биржевое предложение), тренд, безопасность сети.
- **Покрытие on-chain — индикатор, а не гейт.** В шапке отдельно показано, какая доля on-chain-семей
  (MVRV, биржевое предложение, активность, доход майнеров) реально посчитана. Потеря Coin Metrics
  снижает этот показатель — прежде всего убирает биржевое предложение/потоки, — но не создаёт ложный
  оптимизм и не выключает панель: остальные семьи переходят на цельные Blockchain.com-ряды.

## Проверки

### Замороженная live-политика

С 2026-07-19 весь путь решения работает по неизменяемому контракту `btc-decision-suite-v1`:
скоринг, веса, режимы, детекторы, гистерезис, лестница `0/5/20/45/95/100`, recovery floor `80%`,
capitulation floor `40%`, euphoria safety cap `60%` и общие правила исполнения. Сервер формирует
итоговую цель и хэшированный журнал; браузер её только показывает. `backtest/` остаётся
исследовательским архивом и не может менять live-логику. Полные правила — в
[`POLICY.md`](POLICY.md).

Начиная с v2.10.0 снимок ведёт forward/OOS-наблюдение «как было собрано»: policy v1 сравнивается с
прежней policy в shadow-режиме, buy & hold, постоянными 50% BTC и 0% BTC, а также простым
trend/vol-ориентиром. Учитываются оборот, 10 б.п. издержек на полный оборот и доходность cash по
FRED DTB3, предварительно переведённую из discount basis в эффективную годовую доходность по 91-дневной конвенции. Поздние пересмотры данных не переписывают записанные хэши входов и решений. Начиная с v2.11.1 дополнительно проверяются изменения перекрывающегося исторического интервала, даже если источник одновременно добавил новую точку. Публичный снимок хранит проверяемые хэш-обязательства и точные входы итогового решения, но не полные сырые ответы поставщиков.

Независимый монитор проверяет не только свежесть и воспроизводимость `snapshot.json`, но и фактически опубликованные `index.html` и policy-модули, **полную hash-цепь журнала решений** (а не только последнюю запись), регресс `started_at` и непрерывность цепи против закоммиченного состояния. При `decision.quality=paused` или нездоровом операционном статусе интерфейс скрывает процент цели и показывает паузу.

**Инцидент 2026-07-21 и графт.** Утренний откат/рестор policy-слоя молча уничтожил 58-записный
журнал монитора; в опубликованной серии осталось невалидное окно `06:46–07:13 UTC` (ложный
round-trip 20%→5%→20% — не рыночный сигнал). Доинцидентная цепь криптографически провалидирована и
восстановлена графтом (`docs/monitor-graft-v1.json`): перечейненные записи хранят `original_log_hash`,
осиротевшие генезисы перечислены в graft-событии, NAV склеен мостом по прежним целям без повторной
цены входа. Мониторинг непрерывен с **2026-07-19**. С этой же даты действуют новые гарантии: молчаливый
генезис при живом previous оставляет `reset_events`, а `input_hash` воспроизводим из опубликованного
снимка (хэши до 2026-07-21 считались по старой схеме с undefined-ключами и извне невоспроизводимы).
Графт-пакет проверяется static-audit на целостность, пока существует; при будущем **намеренном**
новом генезисе монитора удалите `docs/monitor-graft-v1.json` тем же коммитом — иначе каждый снимок
будет носить метку `graft_skipped`, выглядящую как инцидент.

```
npm run policy    # lock-тест полного policy suite v1 и лестницы аллокации
npm run check     # полный набор: юниты, фикстуры, фоллбэки, workflow, статика, регрессия, self-test, негативные
npm run mock      # сборка снимка на синтетических данных
npm run probe     # карта источников: что реально отвечает с этого хоста
npm run verify    # строгая проверка снимка (то же, что гейт публикации)
```
