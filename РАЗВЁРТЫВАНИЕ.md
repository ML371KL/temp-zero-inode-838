# Как запустить дэшборд на GitHub — только через браузер

Результат: `https://ВАШ-ЛОГИН.github.io/ИМЯ-РЕПОЗИТОРИЯ/`

Терминал не нужен. Всего 7 шагов, около 10 минут.

---

## Шаг 0. Распакуйте архив

Распакуйте `btc-21m-dashboard-v2.7.0.zip` в любую папку. Внутри:

```
scripts/          11 файлов .mjs
docs/             index.html, snapshot.json
.github/workflows/snapshot.yml
package.json
README.md
.gitignore
```

**Важно:** папка `.github` начинается с точки, поэтому в проводнике/Finder она **скрыта**.

- **Windows:** Проводник → вкладка «Вид» → галочка «Скрытые элементы».
- **macOS:** в Finder нажмите `Cmd + Shift + .` (точка).

Если не включить показ скрытых файлов, вы просто не увидите `.github` и не сможете её перетащить.
Ниже это обходится другим способом, так что можно и не включать.

---

## Шаг 1. Создайте репозиторий

1. Войдите на [github.com](https://github.com). Нет аккаунта — зарегистрируйтесь, это бесплатно.
2. Справа вверху **`+`** → **New repository**.
3. Заполните:
   - **Repository name:** например `btc-21m` (это имя попадёт в адрес сайта).
   - **Public** — обязательно. GitHub Pages на бесплатном тарифе работает **только с публичными**
     репозиториями.
   - **Add a README file** — галочку **НЕ ставьте**. Свой README вы зальёте на шаге 3.
4. **Create repository**.

---

## Шаг 2. Разрешите Action писать в репозиторий и публиковать сайт

Сделайте это **до** первой загрузки файлов — иначе первый автоматический запуск упадёт.

1. В репозитории: **Settings** (шестерёнка в верхнем меню репозитория, не в меню профиля).
2. Слева: **Actions → General**.
3. Прокрутите до **Workflow permissions**.
4. Выберите **Read and write permissions**.
5. **Save**.

---

## Шаг 3. Включите GitHub Pages — источник «GitHub Actions»

Тоже **до** загрузки файлов: workflow при первом же запуске захочет опубликовать сайт, и окружение
`github-pages` должно уже существовать.

1. **Settings → Pages** (левое меню).
2. **Build and deployment → Source**: выберите **GitHub Actions**.

> **Не выбирайте «Deploy from a branch».** Документация GitHub прямо говорит: коммиты, сделанные
> workflow-процессом через `GITHUB_TOKEN`, **не запускают пересборку Pages**. При этом варианте
> Action был бы зелёным, свежие данные легли бы в репозиторий — а сайт вечно показывал бы старый
> снимок. Наш workflow публикует сайт сам, поэтому источник должен быть **GitHub Actions**.

Ветку и папку здесь выбирать не нужно — при этом источнике их просто нет.

---

## Шаг 4. Загрузите файлы

### 4а. Обычные папки — перетаскиванием

1. Вкладка **Code** → кнопка **Add file** → **Upload files**.
2. Перетащите в окно браузера **папки `scripts` и `docs`** и **файлы `package.json`, `README.md`**.
   Именно папки целиком — GitHub сохранит структуру.
3. Внизу **Commit changes**.

`.gitignore` можно не заливать — он ни на что не влияет.

### 4б. Файл workflow — создайте вручную

Перетаскивание скрытой папки `.github` в браузере срабатывает не всегда. Надёжнее так:

1. **Add file** → **Create new file**.
2. В поле имени введите **ровно это**, вместе со слешами:

   ```
   .github/workflows/snapshot.yml
   ```

   Как только вы наберёте `/`, GitHub сам превратит это в папки — увидите «крошки»
   `.github / workflows / snapshot.yml`.
3. Откройте у себя на компьютере `.github/workflows/snapshot.yml` (Блокнотом или любым
   редактором), скопируйте **всё содержимое** и вставьте в большое поле на GitHub.
4. Внизу **Commit changes**.

> Если у вас не получается открыть скрытую папку — откройте архив штатным архиватором
> (двойной клик по zip), внутри архива скрытые папки видны всегда.

## Шаг 5. Ключи (необязательно)

Панель работает **без единого ключа**. Ключи только ускоряют и обогащают:

| Ключ | Что даёт | Где взять |
|---|---|---|
| `FRED_KEY` | быстрый макро-блок (без него FRED читается через keyless CSV — медленнее, но работает) | [бесплатно, 1 минута](https://fred.stlouisfed.org/docs/api/api_key.html) |
| `CM_API_KEY` | повышает шанс получить биржевые притоки/оттоки | бесплатный тариф Coin Metrics |

Как добавить: **Settings → Secrets and variables → Actions → New repository secret** → имя
(`FRED_KEY`), значение — сам ключ → **Add secret**.

---

## Шаг 6. Запустите сборку

1. Вкладка **Actions**.
2. Если увидите баннер «Workflows aren't being run on this forked repository» или предложение
   включить Actions — нажмите зелёную кнопку подтверждения.
3. Слева выберите **«Обновление Bitcoin-снимка»**.
4. Справа **Run workflow** → **Run workflow**.
5. Ждите 3–7 минут. Обновите страницу — появится запуск с жёлтым кружком, затем зелёная галочка.

Дальше сборка идёт **сама, раз в 8 часов**.

---

## Шаг 7. Откройте сайт

Адрес: **`https://ВАШ-ЛОГИН.github.io/ИМЯ-РЕПОЗИТОРИЯ/`**

Точную ссылку GitHub покажет в двух местах:

- **Settings → Pages** — вверху, «Your site is live at …»;
- на странице самого запуска в **Actions** — в блоке `github-pages` (шаг «Развернуть сайт»).

## Что вы увидите и как это читать

**До первого успешного запуска** страница показывает синюю плашку
«ДЕМОНСТРАЦИОННЫЙ СНИМОК: числа синтетические» — в `docs/snapshot.json` лежат тестовые данные,
чтобы страница не была пустой. После первого запуска плашка исчезает.

**Самое важное после первого запуска** — откройте лог шага **«Диагностика эндпоинтов»**
(Actions → ваш запуск → «Диагностика эндпоинтов»). Это единственная проверка, которую невозможно
провести заранее: она показывает, что реально ответило **именно с вашего раннера**.

Там будет карта источников — по каждому эндпоинту видно, ответил ли он, критичен ли он и что именно
вернул. Внизу — сводка отказов:

```
STATE CRIT HTTP    MS HOST                         NAME · CONTRACT
OK    YES  200   412 api.coinmetrics.io           Coin Metrics MVRV · 1830 rows · MVRV 1.415
OK     no  200   680 api.blockchain.info          blockchain.info MVRV · 365 pts · MVRV 1.402
OK    YES  200   240 api.exchange.coinbase.com    Coinbase daily price · 300 candles · close 68000
FAIL   no  403   190 www.okx.com                  OKX swap · !code 51001

Critical failures: 0
Optional/fallback failures: 1 · OKX swap
Probe is diagnostic; candidate validation and TTL remain authoritative.
```

Как читать: **STATE** — ответил ли эндпоинт; **CRIT = YES** — источник незаменим (у него нет
бесплатного резерва), **no** — есть резерв или он необязателен. Ключевая строка —
`Critical failures` внизу: сколько именно незаменимых источников не ответило. Если геоблокирована
offshore-площадка (чаще всего OKX или Bybit — геоблок для американских IP раннеров GitHub), она
попадёт в `Optional/fallback failures`, и панель это переживёт: краткосрочный вердикт опирается на
реализованную волатильность, посчитанную из собственного ценового ряда, а не на эти биржи. Сама
диагностика никогда не блокирует публикацию — гейтом остаётся строгая проверка кандидата (`verify`).

## Если что-то пошло не так

| Симптом | Причина | Что делать |
|---|---|---|
| Action красный на шаге «Сохранить снимок» | не выданы права на запись | Шаг 2: Workflow permissions → **Read and write** |
| Action красный на шаге «Развернуть сайт» | источник Pages не «GitHub Actions» | Шаг 3 |
| Сайт отдаёт **404** | Pages ещё не включён, либо запуск не дошёл до конца | Шаг 3, затем дождитесь зелёной галочки в Actions |
| Сайт открывается, но **плашка «демо» не уходит** | первый запуск ещё не завершился успехом | Actions → откройте последний запуск, посмотрите, какой шаг красный |
| Во вкладке Actions **пусто** | файл workflow лежит не там | Путь должен быть **строго** `.github/workflows/snapshot.yml` |
| Action красный на шаге **«Изолированные тесты»** | залиты не все файлы | Проверьте: все 11 файлов в `scripts/`, оба в `docs/`, плюс `package.json` и `README.md` — тесты читают их все |
| Данные на сайте не обновляются, хотя Action зелёный | выбрано «Deploy from a branch» | Шаг 3. Это ровно та ловушка, ради которой источник должен быть **GitHub Actions** |

---

## Обновление панели в будущем

**Add file → Upload files**, перетащить изменённые файлы, **Commit changes**. Загрузка чего-либо
в `scripts/` или `docs/index.html` автоматически запускает пересборку и переразвёртывание.
